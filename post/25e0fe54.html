三层架构

在 Java 开发中，特别是在使用 Spring 或 Spring Boot 等框架进行 Web 开发时，Controller、Service 和 Mapper 是实现分层架构的三个关键组件。这种分层设计的好处是能够分离业务逻辑、数据访问和用户请求处理，减少耦合并提高代码的可维护性。

1. Controller 层（表现层 / 控制器层）

  • 作用：负责处理客户端发送的 HTTP 请求，调用 Service 层处理业务逻辑，并将结果返回给客户端。通常会对外提供 REST API 接口。
  • 功能：
    • 接收用户请求。
    • 调用 Service 层执行具体的业务操作。
    • 将结果封装成响应返回给客户端（如 JSON 数据）。
  • 常用注解：
    • @RestController：标识当前类是一个 RESTful 控制器，会将返回结果自动转换为 JSON 格式。
    • @RequestMapping：定义 URL 路径与控制器方法的映射关系。
    • @GetMapping、@PostMapping：分别对应 HTTP 的 GET 和 POST 请求。

2. Service 层（业务逻辑层）

  • 作用：负责处理具体的业务逻辑。Service 层通常不直接与数据库交互，而是调用 Mapper 层进行数据持久化操作。
  • 功能：
    • 处理业务逻辑。
    • 通过调用 Mapper 层或 Repository 层与数据库交互。
    • 可以包含事务处理逻辑（@Transactional 注解）。
  • 常用注解：
    • @Service：标识当前类是一个服务层组件，Spring 会自动将其注册为 Bean。

3. Mapper 层（数据访问层 / 持久化层）

  • 作用：直接与数据库交互，进行 CRUD 操作。Mapper 层使用 MyBatis 或 JPA 之类的 ORM 框架来简化数据访问。
  • 功能：
    • 执行 SQL 语句，与数据库进行交互。
    • 将数据结果映射到 Java 对象（DTO/POJO）。
  • 常用注解：
    • @Mapper（MyBatis 特定）：用于标识该接口是一个 MyBatis 的 Mapper 接口。
    • MyBatis 中会通过 XML 或注解的形式定义 SQL 语句，如 @Select、@Insert、@Update、@Delete。

三层架构的调用关系：

  1. Controller 层接收用户请求，调用 Service 层处理业务逻辑。
  2. Service 层负责业务处理，通常调用 Mapper 层进行数据库操作。
  3. Mapper 层执行具体的数据库操作，返回数据给 Service 层。
  4. Service 层将处理结果返回给 Controller 层，最后由 Controller 层返回响应给用户。

总结：

  • Controller 负责处理 HTTP 请求和响应。

  • Service 负责业务逻辑和事务控制。

  • Mapper 负责与数据库进行交互，执行 SQL 语句。

  • 控制层：@Controller、@RestController、@RequestMapping、@GetMapping、@PostMapping、@PathVariable、@RequestParam 等。

  • 业务层：@Service、@Autowired、@Transactional。

  • 数据层：@Repository、@Component、@Qualifier、@Value。

  • 配置相关：@Configuration、@Bean、@Profile。

mapper.xml

  • resultType ：主要针对从数据库中提取相应的数据出来 如：select从数据库查询。（sql输出结果类型）

    resultType是sql语句查询结果集的封装类型，也就是说把sql查询的结果封装在bean里返回去，是存数据用的。

  • resultMap：resultType和resultMap功能类似 ，都是返回对象信息 ，但是resultMap要更强大一些 ，可自定义。因为resultMap要配置一下，表和类的一一对应关系，所以说就算你的字段名和你的实体类的属性名不一样也没关系，都会给你映射出来，但是，resultType必须字段名一样，比如说 cId和c_id 这种的都不能映射 。

    resultMap 元素定义了数据库列名和 Java 对象属性名之间的映射关系，以及 address 属性的嵌套映射关系。

    • resultType 和 resultMap 都用于将数据库查询结果映射到 Java 对象
      特性 resultType resultMap
      复杂度 简单 复杂
      灵活度 低 高
      可读性 差 好
      适用场景 简单映射关系 复杂映射关系，嵌套关系
      使用方式 简洁 需要定义 resultMap

      对于简单的映射场景，resultType 足够使用；对于复杂的映射场景，尤其是涉及到嵌套关系、类型转换等情况，resultMap 是更好的选择，因为它提供了更灵活、更强大的映射能力，并且提高了代码的可读性和可维护性。 为了避免潜在的问题，建议优先考虑 resultMap，除非你的映射非常简单且直接。

  • parameterType ：主要针对于将信息存入到数据库中 如：insert 增加数据到数据库。（传入参数类型）

    paramType是从传过来的Bean中取数据放进例如insert语句的values中当实参用，是取数据用的。

  • @Options(useGeneratedKeys = true, keyProperty = "id") 注解用于配置插入操作，使其能够自动获取数据库生成的键值并将其赋值给 Java 对象的对应属性。

    • **useGeneratedKeys = true**: 这个属性告诉 MyBatis 在执行插入语句后，从数据库获取自动生成的键值。 MyBatis 会在执行插入操作后，从数据库获取这个新生成的键值。
    • **keyProperty = "id"**: 这个属性指定 MyBatis 将获取到的自动生成键值赋值给 Java 对象的哪个属性。

    当调用 insertUser 方法插入一个新的用户时，MyBatis 会自动获取数据库生成的 id 值，并将该值赋给 user 对象的 id 属性。 调用方法后，user.getId() 将返回新生成的 id 值。

    • resultMap 定义了如何将查询结果映射到 SetmealVO 对象，以及如何处理嵌套的 SetmealDish 集合。
    • 通过使用 <collection> 标签，你可以将一对多关系中的子表数据（setmeal_dish）映射到主对象（setmeal）的一个集合属性中。
    1. **id="setmealAndDishMap"**：这是 resultMap 的唯一标识符，用于在 MyBatis 查询结果中映射到对应的对象结构。
    2. **type="com.sky.vo.SetmealVO"**：表示这个 resultMap 映射到 SetmealVO 类，SetmealVO 是你的主实体类。
    3. **autoMapping="true"**：表示自动映射功能启用。如果数据库表的字段名和 Java 类的属性名一致，MyBatis 会自动匹配并映射这些字段，而不需要在 resultMap 中显式定义每个字段的映射。
    4. **<result column="id" property="id"/>**：将数据库表中的 id 字段映射到 SetmealVO 类的 id 属性。
    5. **<collection property="setmealDishes" ofType="SetmealDish">**：这是一个集合映射，setmealDishes 是 SetmealVO 中的一个集合属性，类型为 List<SetmealDish>。这段代码的意思是，当查询结果中包含多个 SetmealDish 时，这些结果会被映射到 SetmealVO 的 setmealDishes 属性中。
      • **property="setmealDishes"**：表示 SetmealVO 中的属性名。
      • **ofType="SetmealDish"**：表示这个集合中包含的元素类型是 SetmealDish。

    原本在getByIdWithDish 方法使用 resultType="com.sky.vo.SetmealVO"。这个 SQL 查询通过 LEFT JOIN 获取 setmeal 和相关的 setmeal_dish 数据，因此 setmeal 可能会关联多个 setmeal_dish，导致多条结果被返回。

    当使用的是 resultType，这会将查询结果直接映射到 SetmealVO 对象。如果返回多条记录而期望 selectOne() 只返回一个，这会导致 TooManyResultsException。

vo，pojo，dto

1. VO（Value Object）

VO 全称为 Value Object，通常是用来表示业务场景中的数据对象，代表一个具体的业务实体。它通常用于在 展示层 或 业务层 之间传递数据。

  • 特点：
    • 通常是不可变的，一般是只读的。
    • 一般只包含简单的属性，没有复杂的逻辑或行为。
    • 主要用于传递数据到界面、前端，或表示一组关联的值。
  • 场景：
    • 用于在展示层、前端和后端之间进行数据传输，类似于用户界面上展示的表单数据。
    • 例如，一个用户信息展示对象 UserVO，包含用户的 name、email、age 等属性。

2. DTO（Data Transfer Object）

DTO 全称为 Data Transfer Object，是一种用来在不同 层 之间传输数据的对象，通常用于在 服务层 之间传递数据。DTO 可能会携带一些冗余字段以减少远程调用的次数。

  • 特点：
    • 主要用于传递数据，尤其是在远程调用时使用，常用于 Web 服务、API 调用中。
    • DTO 通常不包含业务逻辑，主要负责传递数据。
    • 可能包含多个实体对象的组合数据。
  • 场景：
    • 在后端服务层中传递复杂的业务数据，减少调用次数。
    • 比如 CategoryDTO 可能会包含分类的 id、type、name、sort 等属性，便于传递分类相关的业务数据。

3. POJO（Plain Old Java Object）

POJO 全称为 Plain Old Java Object，是一个简单的 Java 对象，通常不依赖于任何特殊的框架或库，内部仅包含属性和 getter/setter 方法。

  • 特点：
    • 没有复杂的业务逻辑，不依赖任何 Java EE 或第三方框架。
    • 仅包含字段和基本的 getter/setter 方法，有时也包含 toString、equals 等方法。
    • 它可以是任何一种 Java 对象，VO 和 DTO 也是一种特定的 POJO。
  • 场景：
    • 当你需要一个简单的对象来封装数据时，例如封装数据库查询结果时使用。

4. PO（Persistent Object）

PO 全称为 Persistent Object，主要用于表示数据库中的一条记录。它与数据库中的表结构一一对应，通常在数据持久化层中使用。

  • 特点：
    • 每个 PO 对应数据库中的一张表，每个字段对应表中的一列。
    • 通常是 ORM（如 Hibernate、MyBatis）框架的实体类，用于表示数据库记录。
    • 包含数据库表中的主键和数据字段，但通常不包含业务逻辑。
  • 场景：
    • 主要用于 ORM 框架中，将数据库中的记录映射为 Java 对象。例如 CategoryPO 对应数据库中的分类表 category。

总结
类型 全称 用途 特征
VO Value Object 表示业务层数据，通常用于展示层或业务层传递数据 只读对象，通常表示用户界面上的信息
DTO Data Transfer Object 数据传输对象，用于服务层或 API 调用时传递数据 用于传输数据，可能包含多个实体组合，通常无业务逻辑
POJO Plain Old Java Object 简单的 Java 对象，不依赖于框架 仅包含属性、getter/setter 方法
PO Persistent Object 持久化对象，与数据库中的表结构一一对应 表示数据库中的一条记录，通常与 ORM 框架结合使用

总结来说，VO、DTO、POJO 和 PO 都是为了在不同场景下传递数据或封装数据而定义的不同类型对象，它们的主要区别在于使用的场景和所承担的责任。
