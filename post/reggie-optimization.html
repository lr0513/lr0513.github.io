UserController

这段代码是一个 Spring Framework 的控制器类，用于管理用户（User）相关的请求。它演示了如何使用 Spring 的缓存注解来处理缓存逻辑。以下是对控制器中每个方法的解释：

  1. save 方法：
    • 使用 @CachePut 注解，该方法在创建新用户后，将用户对象放入名为 userCache 的缓存中，键为用户 ID。
    • 通过 userService.save(user) 将用户数据保存到数据库。
  2. delete 方法：
    • 使用 @CacheEvict 注解，该方法在删除用户后，从 userCache 缓存中移除对应 ID 的用户对象。
    • 通过 userService.removeById(id) 根据 ID 删除数据库中的用户。
  3. update 方法：
    • 使用 @CacheEvict 注解，该方法在更新用户后，从 userCache 缓存中移除原用户对象，然后再次将更新后的用户对象放入缓存。
    • 通过 userService.updateById(user) 更新数据库中的用户数据。
  4. getById 方法：
    • 使用 @Cacheable 注解，该方法首先检查 userCache 缓存中是否存在对应 ID 的用户对象。如果存在，则直接从缓存中获取；如果不存在，则调用 userService.getById(id) 从数据库中获取用户数据，并将其放入缓存。
    • unless = "#result == null" condition满足条件时才缓存数据，如果方法返回值为 null，则不进行缓存。或者unless满足条件则不缓存
  5. list 方法：
    • 使用 @Cacheable 注解，该方法根据传入的 User 对象构造查询条件，查询用户列表，并将结果放入名为 userCache 的缓存中。
    • 缓存的键由用户 ID 和用户名组成，这意味着不同的查询条件将有不同的缓存键。
    • 通过 userService.list(queryWrapper) 获取用户列表。

缓存注解详解：

  • @Cacheable：方法执行前，先检查缓存，如果有缓存则直接返回，否则执行方法并将结果存入缓存。
  • @CachePut：方法执行后，将返回值放入缓存，替换掉原有的缓存值。
  • @CacheEvict：清理缓存，可以指定删除某个键或全部缓存。

注意事项：

  • value 或 ""：指定缓存管理器的名称（字符串数组），可以有多个缓存管理器，Spring Cache 会根据这个名称找到对应的缓存管理器进行操作。
  • 缓存键 key 定义了缓存数据的键，可以是方法参数的一部分或整个参数，用于唯一标识存储在缓存中的项的值。使用 SpEL 表达式动态存储，
  • 缓存注解可以有多个参数，如 condition（条件）和 unless（除非），用于更细粒度的控制缓存行为。

这个控制器类展示了如何在不同的 CRUD 操作中使用 Spring Cache 来提高应用程序的性能，通过减少对数据库的直接访问，利用缓存来存储和检索数据。

SpEL（Spring Expression Language）是 Spring Framework 提供的一种强大的表达式语言，用于在运行时查询和操作对象图。SpEL 支持在 Spring 应用程序中以一种声明式的方式访问和修改应用程序的上下文和对象。

当 getById 方法使用 @Cacheable 注解并设置 condition 属性时，如果查询一个不存在的 ID，可能会发生缓存穿透

缓存穿透：访问的数据既不在缓存也不在数据库中

为什么不使用 condition 来处理缓存穿透？

  1. **condition** 的行为：condition 属性是用于确定是否缓存方法的结果。如果 condition 表达式的结果是 false，方法的结果将不会被缓存。但是，这并不能防止在数据库查询返回 null 时缓存 null 值，因为只有当 condition 表达式求值为 false 时，才会阻止缓存。
  2. **unless** 的适用性：unless 属性更适合处理缓存穿透的情况。unless 属性允许您指定一个条件，当该条件为真时，即使方法执行了，其结果也不会被缓存。在您的示例中，unless = "#result == null" 正确地防止了 null 结果被缓存，从而避免了缓存穿透问题。

出现缓存穿透的情况：

  1. 查询不存在的数据：当应用程序查询数据库中不存在的数据（例如，数据库中没有ID为-1的记录），并且这个查询请求没有被缓存，每次都会直接打到数据库上。
  2. 高并发请求：在高并发场景下，如果大量请求都查询同一个不存在的数据，会导致数据库在很短时间内收到大量相同请求，从而影响性能。
  3. 恶意攻击：攻击者可能利用缓存穿透的特性，通过发送大量查询不存在数据的请求，达到对后端数据库进行拒绝服务攻击（DoS攻击）的目的。
  4. 数据更新频繁：当数据更新很频繁，而缓存没有及时更新或同步，可能导致缓存中的数据与数据库中的数据不一致，从而引发缓存穿透。
  5. 缓存策略不当：如果缓存策略设置不当，例如，将所有查询请求的结果都缓存，而没有考虑数据的实际存在性，也可能导致缓存穿透。

处理缓存穿透的正确方式

为了避免缓存穿透，您可以使用以下策略：

  • 使用 **unless** 属性：如您的示例所示，使用 unless = "#result == null" 来防止 null 值被缓存。
  • 预加载空值：在系统启动或定期任务中，预先查询可能被请求的键，并主动将它们的结果（即使是 null）存入缓存，这样即使查询返回 null，请求也会从缓存中返回，而不是每次都查询数据库。
  • 使用布隆过滤器：布隆过滤器可以用来快速判断一个键是否可能不存在数据库中，从而避免对数据库的查询。

MySQL主从复制

MySQL 主从复制是一种用于数据高可用性和数据冗余的技术，它可以将一个数据库服务器（主库 Master）上的数据变更同步到一个或多个数据库服务器（从库 Slave）。以下是您提供的复制过程描述的修正和补充：

![[/Untitled 2 111.png|Untitled 2 111.png]]

写主库，读从库

  1. Master 将变更记录到二进制日志（Binary Log）：
    • 主库上的所有数据变更（如 INSERT、UPDATE、DELETE）都会被记录到二进制日志中。这是复制的基础。
  2. Slave 将 Master 的 Binary Log 拷贝到它的中继日志（Relay Log）：
    • 从库通过一个称为 I/O Thread 的工作线程连接到主库，请求并接收二进制日志文件，并将其内容写入到自己的中继日志中。
  3. Slave 重做中继日志中的事件，将变更应用到自己的数据库中：
    • 从库使用 SQL Thread 来解析中继日志中的事件，并在本地执行这些事件，从而将主库的变更应用到自己的数据库中。

主从复制的线程：

  • I/O Thread：负责在从库上执行网络通信，从主库获取二进制日志的变更。
  • SQL Thread：负责在从库上执行解析和应用中继日志中的事件。

主从复制的类型：

  • 异步复制（Asynchronous Replication）：主库在执行完客户端提交的事务后立即响应，不等待从库的确认，因此可能存在主从延迟。
  • 半同步复制（Semi-synchronous Replication）：主库在执行完事务后会等待至少一个从库确认接收到事务，提高了数据一致性，但牺牲了一定的性能。
  • 组复制（Group Replication）：一种提供高可用性的复制方式，支持多点写入和自动故障转移。

注意事项：

  • 主从复制可以用于读写分离，提高数据库的读性能。
  • 主从复制可以用于数据备份和灾难恢复。
  • 主从复制可能会引入延迟，尤其是在高延迟的网络环境中。
  • 主从复制需要正确配置和监控，以确保复制的一致性和性能。

MySQL 主从复制是一个强大的功能，可以提高数据库的可用性和扩展性。通过合理配置和使用，可以有效地实现数据的同步和保护。

Nginx配置文件结构

  • 全局块：从开始到 events 之前。和Nginx运行相关的全局配置，通常包含了影响整个 Nginx 服务器的指令。这个块没有特定的名称，它一直持续到 events 块之前。全局块中常见的指令包括配置文件的包含指令（include）、定义运行Nginx的用户的指令（user）、工作目录（pid）、日志文件的位置（error_log）等。

  • Events块：和网络连接相关的配置。定义了影响服务器或与客户端连接相关的配置。它通常包括工作进程的连接数、多线程配置、事件驱动模型等设置。

  • Http块：代理、缓存、日志记录、虚拟主机配置。这个块是配置文件中最重要的部分之一，它包含了定义如何处理HTTP请求的指令。Http块可以包含一个或多个服务器块（server block），每个服务器块定义了特定的虚拟主机或服务器的配置。

    • http全局块
    • Server块
      • Server全局块
      • location块

    注意：http块中可以配置多个Server块，每个Server块中可以配置多个location块。

每个服务器块（server block）可以包含以下部分：

  • Listen指令：指定服务器监听的端口和协议。
  • Server_name指令：定义当前服务器块响应的域名。
  • Location块：定义请求的处理规则，可以根据URL路径、正则表达式等匹配请求，并定义相应的处理方法，如代理设置、重定向、静态文件服务等。

Nginx具体应用

  1. 部署静态资源

    Nginx可以作为静态web服务器来部署静态资源。静态资源指在服务端真实存在并且能够直接展示的一些文件，比如常见的html页面、css文件、js文件、图片、视频等资源。相对于Tomcat，Nginx处理静态资源的能力更加高效，所以在生产环境下，一般都会将静态资源部署到Nginx中。将静态资源部署到Nginx非常简单，只需要将文件复制到Nginx安装目录下的html目录中即可。

cp hello.html /usr/local/nginx/html

http://192.168.56.101/hello.html

  1. 反向代理

正向代理：

是一个位于客户端和原始服务器(origin server)之间的服务器，为了从原始服务器取得内容，客户端向代理发送一请求并指定目标(原始服务器)，然后代理向原始服务器转交请求并将获得的内容返回给客户端。正向代理的典型用途是为在防火墙内的局域网客户端提供访问Internet的途径。正向代理一般是在客户端设置代理服务器，通过代理服务器转发请求，最终访问到目标服务器

![[/Untitled 9 65.png|Untitled 9 65.png]]

反向代理服务器位于用户与目标服务器之间，但是对于用户而言，反向代理服务器就相当于目标服务器，即用户直接访问反向代理服务器就可以获得目标服务器的资源，反向代理服务器负责将请求转发给目标服务器。用户不需要知道目标服务器的地址，也无须在用户端作任何设定。

Nginx 可以接收客户端的请求并转发到后端服务器上，然后再将后端服务器的响应转发回客户端。

![[/Untitled 10 61.png|Untitled 10 61.png]]

  • 举个例子
    • 正向代理：你让舍友去给你带三楼卖的煎饼（你最终会得到一个三楼的煎饼）
    • 反向代理：你让舍友去给你买煎饼（你最终只会得到一个煎饼，但你不知道煎饼是哪儿卖的）
    • 和正向代理不同，反向代理相当于是为目标服务器工作的，当你去访问某个网站时，你以为你访问问的是目标服务器，其实不然，当你访问时，其实是由一个代理服务器去接收你的请求，正向代理与反向代理最简单的区别： 正向代理隐藏的是用户（卖煎饼的不知道是你要买），反向代理隐藏的是服务器（你不知道煎饼是谁卖的）。
    • 正向代理侧重的是用户，用户知道可以通过代理访问无法访问的资源，而反向代理侧重点在服务器这边，用户压根不知道自己访问的是资源时通过代理人去转发的。
  • 客户端请求:
    • 客户端（浏览器或其他 HTTP 客户端）向 NGINX 服务器的 82 端口发送 HTTP 请求。例如，客户端访问 http://localhost:82/。
  • NGINX 接收请求:
    • NGINX 在 82 端口监听到客户端的请求，且请求的域名是 localhost，因此它匹配到上面的 server 配置。
  • 请求转发到后端:
    • 根据 location / 的配置，NGINX 会将客户端请求转发到 proxy_pass 指定的地址，即 http://192.168.56.1:8080。
    • 也就是说，原本发往 localhost:82 的请求，NGINX 会在内部通过代理转发到 192.168.56.1:8080 服务器。
  • 后端响应:
    • 后端服务器（在这里是 192.168.56.1:8080）接收到请求后，会处理并返回一个 HTTP 响应，可能是一个 HTML 页面、JSON 数据或其他类型的内容。
  • NGINX 转发响应:
    • 后端服务器处理完成并返回响应后，NGINX 会接收到这个响应，并将其转发回客户端。
    • 客户端收到的就是后端服务器的响应内容，但这个响应看起来像是来自 NGINX 服务器，因为请求经过了反向代理。

1. 什么是反向代理？

反向代理的主要目的是隐藏真实服务器的地址，处理来自客户端的请求，然后将请求转发到内部服务器上，最后将服务器的响应返回给客户端。

  • 工作流程：客户端 -> 反向代理服务器 -> 内部服务器 -> 反向代理服务器 -> 客户端
  • 主要功能：
    • 隐藏内部服务器的IP地址和端口，增强安全性。
    • 提供SSL加密等功能。
    • 缓存静态资源，减少服务器压力，提升响应速度。

例如，将https://example.com的请求转发到实际的服务器http://localhost:8080。客户端并不知道真实服务器的地址，而只与Nginx交互。

2. 什么是负载均衡？

负载均衡的主要目标是分发请求到多个服务器，以减轻单台服务器的负载压力，提升系统的整体处理能力和可靠性。

  • 工作流程：客户端 -> 负载均衡服务器 -> 服务器A/B/C… -> 负载均衡服务器 -> 客户端
  • 主要功能：
    • 根据策略（如轮询、最小连接数、IP哈希等），分发请求到不同服务器上。
    • 当某台服务器故障时，可以自动将请求分配给其他可用服务器，提高可用性。

例如，当用户访问Nginx服务器时，Nginx根据配置的策略将请求分发给端口为8081、8082的多台服务器，来减轻每台服务器的负担。

Nginx中反向代理和负载均衡的关系

在Nginx中，反向代理和负载均衡常常一起使用，即Nginx既充当反向代理服务器，又充当负载均衡服务器：

  • 反向代理部分：Nginx接收客户端请求，转发请求，并返回响应。
  • 负载均衡部分：Nginx将请求分发给不同的服务器，以提高系统的吞吐量和容错性。

命令 src/redis-server ./redis.conf 用于启动 Redis 服务器，并使用当前目录下的 redis.conf 配置文件来配置 Redis 实例。下面是命令各部分的详细解释：

  • src/redis-server：这是从 Redis 源代码编译后生成的 Redis 服务器可执行文件的路径。通常，Redis 服务器的可执行文件会位于源代码目录的 src 子目录中。
  • ./redis.conf：这是 Redis 配置文件的路径。./ 表示当前目录，redis.conf 是配置文件的名称。这个配置文件包含了启动 Redis 服务器时所需的各种设置，例如持久化选项、网络配置、密码保护、日志记录等。

综合来看，执行这个命令会启动 Redis 服务器，并且使用当前目录下的 redis.conf 文件作为配置源。这意味着 Redis 服务器将根据 redis.conf 文件中定义的参数来初始化和运行。
