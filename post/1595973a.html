HDFS的Shell操作

进程启停管理

Hadoop HDFS组件内置了HDFS集群的一键启停脚本。

  • $HADOOP_HOME/sbin/start-dfs.sh，一键启动HDFS集群
    执行原理：
    • 在执行此脚本的机器上，启动SecondaryNameNode
    • 读取core-site.xml内容（fs.defaultFS项），确认NameNode所在机器，启动NameNode
    • 读取workers内容，确认DataNode所在机器，启动全部DataNode
  • $HADOOP_HOME/sbin/stop-dfs.sh，一键关闭HDFS集群
    执行原理：
    • 在执行此脚本的机器上，关闭SecondaryNameNode
    • 读取core-site.xml内容（fs.defaultFS项），确认NameNode所在机器，关闭NameNode
    • 读取workers内容，确认DataNode所在机器，关闭全部NameNode
      除了一键启停外，也可以单独控制进程的启停。
  1. $HADOOP_HOME/sbin/hadoop-daemon.sh，此脚本可以单独控制所在机器的进程的启停
    用法：hadoop-daemon.sh (start|status|stop) (namenode|secondarynamenode|datanode)
  2. $HADOOP_HOME/bin/hdfs，此程序也可以用以单独控制所在机器的进程的启停
    用法：hdfs --daemon (start|status|stop) (namenode|secondarynamenode|datanode)

ps：普通用户hadoop默认是shell，无法使用交互式操作，修改成bash：chsh -s /bin/bash

文件系统操作命令

https://hadoop.apache.org/docs/r3.3.4/hadoop-project-dist/hadoop-common/FileSystemShell.html

  1. 命令操作：hadoop fs 和 hdfs dfs
    在 Hadoop 中，文件系统操作有两个命令体系，分别是：
    • hadoop fs：这是 旧版命令，在早期的 Hadoop 版本中使用。你可以使用这个命令来对 HDFS 或本地文件系统进行操作。
    • hdfs dfs：这是 新版命令，目前大多数用户都会使用 hdfs dfs 来执行 HDFS 文件系统操作。它提供了与 hadoop fs 相同的功能，但命令格式更加统一和规范。
  2. 文件系统协议：file:// 和 hdfs://
    • file://：表示 Linux 本地文件系统。当操作本地文件系统时，路径前缀为 file://。
    • hdfs://namenode_server:port/：表示 HDFS 文件系统，用于指定 HDFS 集群的 NameNode 地址及端口号，通常路径前缀为 hdfs://。

    注意：一般情况下，file:// 和 hdfs:// 的协议头可以省略，系统会自动根据路径来识别。如果你明确需要使用某个协议头，最好加上，不然可能会出现路径识别错误。

  1. 常用命令
    • mkdir：创建目录
      • hdfs dfs -mkdir [-p] <path>
        -p选项的行为与Linux mkdir -p一致，会创建父目录
    • ls：列出目录内容
      • -h：以人类可读格式显示文件大小（如 KB、MB）
      • -R：递归列出目录内容
    • cat：查看文件内容
    • cp：复制文件或目录
      • f：强制覆盖目标文件
      注：
      • hdfs dfs -cp：用于在 HDFS 与 HDFS 之间复制文件。
      • hdfs dfs -put：用于将 本地文件 上传到 HDFS。
      • hdfs dfs -get：用于将 HDFS 文件 下载到 本地文件系统
    • mv：移动或重命名文件 注：hdfs内部移动
    • rm：删除文件或目录
      • r：递归删除目录
      • skipTrash：跳过回收站，直接删除
    • put：上传文件到HDFS
      • 语法：hdfs dfs -put [-f] [-p] <localsrc>... <dst>
        • -f：覆盖已存在文件
        • -p：保留文件属性
        • localsrc：本地路径（客户端机器）
        • dst：HDFS路径
      注：vim是在linux中创建，cat也是本地
      如果不加协议头，系统默认会将前一个视为本地文件
      linux使用
      ls，hdfs使用hdfs dfs -ls
    • get：从 HDFS 下载文件到本地文件系统
      • `hdfs dfs -get [-f] [-p] …
        • 下载文件到本地文件系统指定目录，localdst必须是目录
        • -f 覆盖目标文件（已存在下）
        • -p 保留访问和修改时间，所有权和权限。
    • appendToFile：向文件末尾追加内容
      • 语法：hdfs dfs -appendToFile <localsrc> ... <dst>
        • 支持将 本地文件/标准输入 内容追加到 HDFS 文件
        • 如果 <localsrc> 为 -，则从标准输入读取
      注：HDFS 文件修改仅支持 追加 和 删除（不能直接修改已有内容）
    • appendToFile：向文件末尾追加内容
      • hdfs dfs -appendToFile <localsrc> ... <dst>
        • 将所有给定本地文件/标准输入的内容追加到已存在的HDFS文件dst中
        • dst如果文件不存在，将创建该文件。
        • 如果<localSrc>为-，则输入为从标准输入中读取。
           注：hdfs系统的文件修改只支持 ==追加和删除==
  2. HDFS WebUI 操作

HDFS权限

HDFS 超级用户及权限控制
在 HDFS 中，文件的权限控制逻辑与 Linux 文件系统非常相似，主要有 读（r）、写（w） 和 执行（x） 权限，分别对应不同的用户组。可以通过 r、w、x 来控制文件或目录的访问权限，权限的设置方式和 Linux 中的权限机制一致。

  • rw-r--r--：表示文件拥有者（Owner）有读写权限（rw），用户组（Group）和其他用户（Other）只有读权限（r）。
  • drwxrwx---：表示目录权限，目录拥有者和用户组都拥有读写执行权限，而其他用户没有权限。
  • rW-r--r：表示文件权限，拥有者有读写权限，用户组有只读权限，而其他用户有读权限，但“W”表示不允许写入权限。
  • rw-r--r：表示文件拥有者有读写权限，用户组和其他用户只有读取权限。
    超级用户：
  • Linux 的超级用户：是 root 用户，拥有系统的所有权限。
  • HDFS 的超级用户：是启动 NameNode 的用户（通常是 hadoop 用户）。hadoop 用户具备在 HDFS 中执行所有操作的权限。

与 Linux 中类似，HDFS 中也可以使用 chown 和 chmod 命令来修改文件或目录
的所有者及权限。

  • 修改文件/目录的所有者和组：
    使用
    chown 命令来修改文件或目录的所有者和组。
    • R：递归修改目录及其子目录的所有者和组。
    • root:root：表示设置文件或目录的所有者为 root，组为 root
  • 修改文件/目录的权限：
    使用
    chmod 命令来修改文件或目录的访问权限。
    • 777：表示设置文件或目录的权限为所有用户可读、可写、可执行。
    • R：递归修改目录及其子目录的权限。

HDFS客户端 - NFS（网络文件系统）

为了让传统的应用程序能够像访问普通文件系统一样访问 HDFS，Hadoop 提供了一个 NFS（Network File System） 接口。通过这个接口，HDFS 可以作为一个标准的文件系统挂载到客户端系统上，用户可以直接通过标准的文件系统协议来读写 HDFS 上的文件。
NFS 网关支持 NFSv3，并允许将 HDFS 作为客户机本地文件系统的一部分挂载，现在支持：上传、下载、删除、追加内容
Hadoop 提供了一个 NFS Gateway，它允许通过 NFS 协议访问 HDFS。NFS Gateway 实际上是一个服务，它将 HDFS 的操作（如读取、写入文件）映射为 NFS 请求，并将这些请求转发到 HDFS。
NFS Gateway 的主要功能是：

  • 将 HDFS 作为 NFS 文件系统挂载：使客户端可以通过 NFS 协议挂载 HDFS。
  • 支持标准 NFS 操作：客户端可以执行标准的 NFS 操作，如读、写、删除文件等。
  • 提供访问控制：通过配置，可以设置哪些客户端可以访问 HDFS，确保数据安全。

要启用 HDFS 的 NFS 服务，必须配置并启动 NFS Gateway。以下是配置步骤。

HDFS存储原理

  • 数据存储的基本结构：
    • HDFS 采用的是 主/从结构，即包含 NameNode（主节点） 和 DataNode（数据节点） 两种角色。
      • NameNode：负责管理整个HDFS的元数据，指引数据的存储和访问，但不存储实际数据。它记录了文件系统的目录结构、文件与块之间的映射关系等信息。
      • DataNode：实际存储数据块的节点，每个DataNode存储一个或多个文件的数据块（Block）。
  • 文件的分块存储：
    • HDFS将大文件分成固定大小的块（默认是128MB），并将这些数据块分布在多个DataNode上进行存储。这种分块存储的方式带来了数据的高可用性、易扩展性和容错性。
    • 例如，文件A被切分为多个数据块：A1, A2, A3… 这些数据块将被分布存储在不同的DataNode上。
  • 数据冗余（副本）：
    • 为了保证数据的高可用性，HDFS会将每个数据块复制多份（默认是3份）。这些副本存储在不同的DataNode上，以便在某个DataNode出现故障时，可以通过其他副本进行数据恢复。
    • 副本的数量是可以配置的，通常为3份，但可以根据需求调整。
  • 写入操作：
    • 在HDFS中，文件是按顺序写入的，一旦写入完成就不能修改（HDFS是一个只追加数据的文件系统），因此对于写操作的优化非常重要。
    • 客户端写入文件时，首先向NameNode请求文件的写入权限，NameNode返回一组DataNode节点的地址（这些节点将用于存储数据块）。然后，客户端将数据流式写入DataNode，DataNode会将数据分块存储，并根据配置的副本数进行复制。
  • 读取操作：
    • 客户端读取文件时，首先向NameNode请求文件的元数据，NameNode会返回该文件的分块位置（即存储哪些DataNode上）。然后客户端直接向这些DataNode请求文件的数据块。
    • 读取操作一般是并行的，客户端从多个DataNode读取不同的数据块，因此可以提高读取速度。
  • 容错与数据恢复：
    • HDFS内建了容错机制。当一个DataNode发生故障时，HDFS会自动通过副本机制恢复数据。NameNode会检测到DataNode失效，然后重新在其他DataNode上创建副本来恢复丢失的数据。
    • 由于数据块的副本分布在不同的DataNode上，即使某些节点发生故障，系统也能保证数据不丢失。
  • 块报告与心跳机制：
    • 心跳机制：DataNode定期向NameNode发送心跳信号，表示它还在正常工作。若NameNode长时间未收到某个DataNode的心跳信号，就会认为该DataNode已失效。
    • 块报告：DataNode会定期向NameNode发送块报告，告诉NameNode它存储了哪些数据块，便于NameNode更新其元数据，保持整个系统的数据一致性。
  • 数据写入与副本的管理：
    • 当文件正在写入时，NameNode会监控每个数据块的副本分布，确保副本分布在不同的DataNode上，以保证高可用性和容错性。
    • 如果某些副本的数量不足（例如某个DataNode故障，导致副本丢失），NameNode会通知其他DataNode进行数据块复制，恢复副本数量。

NameNode元数据

在HDFS中，文件会被划分为多个数据块（Block），那么当文件很大，或者文件数量很多时，Hadoop是如何记录和整理文件和这些数据块之间的关系呢？这个工作正是由 NameNode 完成的。

  1. NameNode的元数据管理
    NameNode负责管理整个HDFS文件系统的元数据。元数据包括文件到数据块的映射关系、块的副本信息等。为了高效地管理这些元数据，HDFS使用了两个关键文件：
    1. Edits 文件：记录每一次对HDFS的操作，类似一个流水账。每次创建、删除、重命名文件、修改文件的副本数等操作都会记录到此文件中。这个文件包含了所有操作对HDFS文件系统的更改。
    2. FSImage 文件：是HDFS的元数据镜像。它保存了一个完整的文件系统的快照，包括文件和数据块的映射关系。
  2. Edits与FSImage的配合
    • Edits 文件记录了HDFS的每一个操作，随着时间的推移，它会变得越来越大。为了保证检索效率，HDFS会定期将多个Edits文件合并生成新的FSImage文件。
    • 当用户访问一个文件时，NameNode需要从Edits文件中读取所有相关的操作记录，按顺序检索到最后，以便恢复到当前的文件系统状态。由于Edits文件可能非常大，这会造成查询效率低下，因此需要合并操作。
  3. 合并过程
    1. 定期合并Edits文件：
      • 每次HDFS发生操作时，Edits文件会记录这些操作，直到达到某个阈值。
      • 一旦Edits文件过大，系统会将其与FSImage文件合并，生成新的FSImage文件。
    2. FSImage的生成：
      • 如果当前没有FSImage文件，HDFS会将所有的Edits合并为第一个FSImage文件。
      • 如果已经存在FSImage文件，则会将所有Edits文件和当前的FSImage文件合并，形成新的FSImage文件。
  4. 元数据合并控制参数
    HDFS有一些参数用于控制Edits文件的合并和FSImage的生成：
    1. dfs.namenode.checkpoint.period（默认3600秒，1小时）：
      • 指定HDFS NameNode合并Edits文件和生成新的FSImage的时间周期。
    2. dfs.namenode.checkpoint.txns（默认1000000，100万次事务）：
      • 指定达到一定数量的操作后，触发Edits文件的合并。
    3. dfs.namenode.checkpoint.check.period（默认60秒）：
      • 指定检查是否达到合并条件的周期，默认每隔60秒检查一次。
  5. SecondaryNameNode的作用
    对于Edits文件和FSImage文件的合并，HDFS有一个辅助角色——SecondaryNameNode。它的主要任务就是定期从NameNode拉取Edits和FSImage文件，并进行合并操作：
    • 合并过程：SecondaryNameNode会拉取当前的FSImage文件和Edits文件，合并后生成新的FSImage文件。
    • 提供合并结果：合并后的新FSImage文件会传回给NameNode，更新HDFS的元数据。

    注意：SecondaryNameNode并不是NameNode的备份，它只是负责元数据的合并工作，确保NameNode不会因为Edits文件过大而导致性能问题。
    总结

  6. Edits文件：记录所有的HDFS操作，并且逐渐增大。
  7. FSImage文件：是HDFS文件系统元数据的快照。
  8. 合并操作：HDFS会定期将Edits文件合并到FSImage文件中，减少Edits文件的大小，并确保NameNode可以快速恢复。
  9. SecondaryNameNode：负责从NameNode拉取Edits和FSImage，执行合并操作并返回合并后的结果。

HDFS数据的读写流程

HDFS（Hadoop Distributed File System）设计的目标之一是提供高效、可靠、可扩展的存储和数据访问方式。以下是HDFS中数据的读写流程以及其关键点。
写数据流程

  1. 客户端向NameNode发起请求：
    客户端首先向NameNode请求写数据操作，NameNode会审核请求是否合法，如用户是否有写权限、集群是否有足够的存储空间等。
  2. NameNode允许写入并返回DataNode的地址：
    NameNode审核完毕后，如果条件满足，允许客户端进行写入。随后，NameNode会返回一组DataNode的地址，告诉客户端将数据写入哪些DataNode上。
  3. 客户端向指定的DataNode发送数据包：
    客户端将数据发送给其中一个指定的DataNode，这个DataNode一般是离客户端最近（网络距离最短）的节点。
  4. DataNode完成数据副本的复制工作：
    被客户端写入数据的DataNode（例如DataNode1）会同时负责数据副本的复制工作。该DataNode接收到数据后，会按照HDFS的副本机制，将数据分发给其他DataNode。
  5. 数据副本的分发：
    比如，DataNode1将数据复制到DataNode2，然后DataNode2将其数据复制到DataNode3和DataNode4。此过程通过构建一个DataNode间的流水线（Pipeline）完成。
  6. 写入完成，客户端通知NameNode：
    客户端完成数据写入后，会向NameNode发送通知，告知数据写入完成。此时，NameNode会更新文件的元数据，以记录新写入数据的块信息。
    关键信息点：
  • NameNode不负责数据的写入：NameNode只负责元数据的记录、权限审核和调度，不直接参与数据的传输。数据的实际写入和副本复制是由DataNode之间协作完成的。
  • 客户端直连DataNode写数据：客户端会根据NameNode返回的DataNode信息直接向离其最近的DataNode发送数据。
  • 副本复制由DataNode完成：数据块的副本复制是由DataNode之间自发地完成，形成一个数据传输流水线，确保数据副本的分布。
    读数据流程
  1. 客户端向NameNode申请读取文件：
    客户端向NameNode发起读取请求，NameNode在审核完权限和其他条件后，返回该文件的所有数据块的列表（Block列表）。
  2. NameNode返回文件的block列表：
    NameNode根据文件的块信息，返回一个块列表。每个文件都会分成多个块，每个块有多个副本，NameNode会返回这些块的信息以及它们的存储位置。
  3. 客户端根据块信息直接向DataNode读取数据：
    客户端通过查询NameNode得到的块列表后，直接连接相应的DataNode读取数据块。数据读取时，客户端会优先选择离其网络距离最近的DataNode进行数据读取。

关键信息点：

  • 数据通过DataNode直接传输，不经过NameNode：NameNode仅提供文件的元数据和块的分布信息，数据的实际读取是由客户端和DataNode直接进行的。
  • 客户端获取的块列表会考虑网络距离：当文件的块有多个副本时，NameNode会优先提供离客户端网络距离最近的副本地址。这可以通过HDFS的内置网络距离计算算法来确定，基于IP地址和路由信息进行距离评估。

注：客户端可以是hadoop fs、hdfs dfs、Big Data Tools插件
